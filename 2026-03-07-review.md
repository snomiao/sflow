# sflow Comprehensive Review — 2026-03-07

## 1. Purpose & Vision

**sflow** is a TypeScript stream-processing library built on the native WebStreams API (`ReadableStream` / `TransformStream`). It provides a **fluent, chainable API** for composing async data pipelines — analogous to RxJS for Observables or Lodash for arrays, but targeting the web-standard streaming primitive.

**Core value proposition:**

- **Async-first**: Every pipeline method accepts async functions, enabling seamless I/O-bound concurrency (API calls, DB queries, file reads) within stream pipelines.
- **WebStreams-native**: No custom stream abstraction — wraps and returns standard `ReadableStream` objects, ensuring interop with `fetch()`, `Response`, service workers, and the broader platform.
- **Functional composition**: Operators are individually exported as TransformStream factories, enabling tree-shakeable, à-la-carte usage alongside the ergonomic chain API.

**Target audience:** TypeScript/JavaScript developers building data pipelines, ETL workflows, real-time event processing, or any scenario where data arrives incrementally and should be transformed without full materialization.

### 2.2 Core Design Patterns

#### FlowSource — Polymorphic Input

```typescript
type FlowSource<T> =
  | Promise<T> | Iterable<T> | AsyncIterable<T>
  | (() => Iterable<T>) | (() => AsyncIterable<T>)
  | ReadableStream<T> | ReadableLike<T>
```

This union type lets `sflow()` accept arrays, generators, promises, async generators, and raw ReadableStreams uniformly. Conversion is handled by `web-streams-extensions`'s `from()`.

#### TransformStream Factory Pattern

Every operator is a function returning `new TransformStream<In, Out>({ transform, flush })`. This enables two usage modes:

```typescript
// Chain API
sflow([1,2,3]).map(x => x * 2).filter(x => x > 2).toArray()

// Standalone composition
stream.pipeThrough(maps(x => x * 2)).pipeThrough(filters(x => x > 2))
```

#### Conditional Type Extensions

The `sflow()` return type dynamically includes extra methods based on the item type:

- `T extends any[]` → `.flat()`, array operations
- `T extends Record` → `.unwind()`, `.mapAddField()`, `.mapMixin()`
- `T extends string` → `.lines()`, `.match()`, `.replace()`
- `T extends ReadableStream` → `.confluence()` and variants
- `T extends string | Uint8Array` → `.toResponse()`, `.text()`, `.json()`

This provides **type-safe method availability** without runtime overhead.

#### Concurrency Management

`map()`, `forEach()`, `pMap()`, and `asyncMap()` accept `{ concurrency?: number }`:

| Function | Order | Default Concurrency |
|----------|-------|-------------------|
| `map(fn, {concurrency})` | Preserved | 1 (sequential) |
| `pMap(fn, {concurrency})` | Preserved | **Infinity** |
| `asyncMap(fn, {concurrency})` | Completion order | **Infinity** |
| `forEach(fn, {concurrency})` | Preserved | 1 (sequential) |

### 2.3 Key Implementation Details

**mergeStream**: Fan-in via `Promise.all` — each source writes to a shared `WritableStream` concurrently. No buffering; backpressure respected per-source.

**mergeAscends**: N-way sorted merge using slot-based coordination. Each source fills a slot; `rambda.sortBy()` selects the minimum. Validates ordering invariant at every emission.

**throttle**: Two modes — backpressure-based (pauses upstream via `await`) and drop-based (discards intermediate chunks, optionally keeping last).

**debounce**: Classic timer-reset pattern, emitting only when the quiet period elapses.

**cacheLists / cacheTails**: Accumulate stream data in-memory arrays attached to a `keyv`-compatible cache store.

## 3. Ideas & Design Philosophy

### What sflow gets right

1. **WebStreams as the substrate.** Rather than inventing a custom Observable/Stream type, sflow wraps the platform primitive. This means interop with `fetch`, `Response`, `TransformStream`, file APIs, and any future platform additions is essentially free.

2. **Dual API surface.** The fluent chain API (`sflow([...]).map().filter()`) is ergonomic for casual use. The standalone operator exports (`maps`, `filters`, `chunks`) are composable and tree-shakeable for library authors.

3. **Async-first design.** Every operator transparently handles `async` callbacks with optional concurrency control — this is genuinely useful for I/O-bound pipelines and distinguishes sflow from synchronous alternatives.

4. **Small, focused modules.** The average 37-line module size makes the codebase highly navigable. Each operator can be understood in isolation.

5. **Strong type-level ergonomics.** Conditional types provide contextual method availability without runtime cost. The `FlowSource` union makes the API forgiving about input types.

### Areas for consideration

1. **The "everything-on-one-object" pattern.** The `sflow()` factory creates an object with 80+ methods. This is convenient but means the chain API cannot be tree-shaken — importing `sflow` pulls every operator into the bundle.

2. **`Infinity` as default concurrency** for `pMap` and `asyncMap`. This means these operations buffer the **entire stream** in memory by default. Users unfamiliar with this default may encounter OOM on large streams. A finite default (e.g., 16) with explicit `Infinity` opt-in would be safer.

3. **Heavy dependencies for minimal usage.** `rambda` (1.8 MB install) is imported for just `sortBy` and `equals`. `d3` (4.6 MB across 62 packages) is used only for CSV parsing via `d3-dsv`. These could be replaced with lightweight alternatives or inline implementations.

4. **Ordered merge uses sort instead of heap.** `mergeAscends` calls `sortBy()` on all candidates per emission — O(N log N) where N is the number of sources. A min-heap would reduce this to O(log N).

## 4. Complexity Analysis

### 4.1 Time Complexity

| Operation | Per-item Cost | Notes |
|-----------|--------------|-------|
| `map(fn)` | O(1) + fn cost | Direct pass-through |
| `filter(fn)` | O(1) + fn cost | |
| `flatMap(fn)` | O(k) where k = output items | |
| `flat()` | O(k) where k = array length | |
| `chunk(n)` | O(1) amortized | Emits every n items |
| `reduce(fn, init)` | O(1) + fn cost | |
| `uniq()` | O(1) average | Set.has() lookup |
| `throttle(ms)` | O(1) | Timer-based |
| `debounce(ms)` | O(1) | Timer-based |
| `mergeStream(N sources)` | O(1) per item | Fan-in, no sorting |
| `mergeAscends(N sources)` | **O(N log N)** per item | `sortBy` on all candidates |
| `mergeStreamsBy(N sources)` | Depends on user `transform` fn | |
| `pMap(fn, {concurrency: C})` | O(1) amortized | Buffers up to C promises |
| `skip(n)` | O(1) | Counter decrement |
| `takeWhile(fn)` | O(1) + fn cost | |

### 4.2 Space Complexity

#### O(1) — True streaming (no buffering)

`map`, `filter`, `flatMap`, `flat`, `skip`, `limit`, `takeWhile`, `peek`, `forEach` (sequential), `throttle`, `debounce`, `riffle`, `mergeStream`

#### O(n) — Bounded buffering

| Operation | Buffer Size |
|-----------|-------------|
| `chunk(n)` | n items |
| `chunkOverlaps(size, step)` | size items |
| `convolve(n)` | n items (sliding window) |
| `head(n)` | n items |
| `map(fn, {concurrency: C})` | C items |
| `pMap(fn, {concurrency: C})` | C promises |
| `asyncMap(fn, {concurrency: C})` | C promises |
| `mergeAscends(N sources)` | N slots |

#### O(stream) — Potentially unbounded

| Operation | What grows | Risk |
|-----------|-----------|------|
| `tail(n)` | Consumes entire stream, keeps last n | Must read all data |
| `uniqs()` | Set of all seen items | Grows with distinct count |
| `cacheLists()` | Array of all items | Full materialization |
| `cacheTails()` | Array of all items | Full materialization |
| `toArray()` | Result array | Terminal, expected |
| `pMaps(fn)` (default) | All items as promises | Default concurrency = Infinity |
| `asyncMaps(fn)` (default) | All items as promises | Default concurrency = Infinity |
| `chunks()` (no arg) | All items | Effectively `toArray()` in a transform |
| `chunkIntervals(ms)` | Items during interval | Unbounded if source is fast |

### 5.5 Runtime Performance Considerations

**Backpressure.** sflow inherits WebStreams' built-in backpressure mechanism. Most operators use `await ctrl.enqueue()` or `await writer.write()`, which naturally pauses the upstream when the downstream is slow. The `highWaterMark` defaults (typically 1) keep memory usage bounded in standard pipelines.

**Microtask overhead.** Each `transform` call goes through an `async` function boundary even for synchronous operations. The `maps()` implementation optimizes this by checking `ret instanceof Promise` before awaiting, avoiding an unnecessary microtask for sync mappers.

**Merge performance.** For unordered merges, performance is O(1) per item (fan-in). For ordered merges (`mergeAscends`), the O(N log N) sort per emission could be a bottleneck with many sources — a heap-based approach would be O(log N).

## 6. Strengths

1. **Tiny own-code footprint** — 10 KB gzipped for 50+ stream operators
2. **WebStreams-native** — interops with platform APIs without adapters
3. **Async-first** with configurable concurrency on map/forEach operations
4. **Rich operator library** — chunking, merging, throttling, debouncing, caching, text processing, object transforms
5. **Strong TypeScript types** — conditional method availability, generic constraints, FlowSource polymorphism
6. **Good test coverage** — 1.3x test-to-source ratio, 198 passing tests
7. **Clean module architecture** — single-responsibility modules averaging 37 lines
8. **Dual API** — chain API for ergonomics, standalone exports for composition

## 7. Potential Improvements

1. **Move `bson`, `npm-run-all`, `ts-essentials` to devDependencies** — they are not imported at runtime
2. **Replace `rambda`** (1.8 MB) with inline `sortBy`/`equals` (~20 lines) — only 2 functions are used
3. **Evaluate polyfill necessity** — `TextEncoderStream`/`TextDecoderStream` are native in Node 16+, Bun, and all modern browsers. The `polyfill-text-encoder-stream` package alone is 5.6 MB
4. **Replace `d3` with `d3-dsv`** in the d3 entry point — only CSV/DSV parsing is used, avoiding 62 sub-packages (4.6 MB → ~66 KB)
5. **Change default concurrency** of `pMaps` and `asyncMaps` from `Infinity` to a bounded value (e.g., 16) to prevent accidental full-stream buffering
6. **Use a min-heap** in `mergeAscends` instead of `sortBy` for O(log N) per-emission instead of O(N log N)
7. **Consider code-splitting** the ESM output per-operator to enable full tree-shaking when using the chain API

---

## 8. Summary

sflow is a **well-designed, well-tested stream processing library** that makes a smart bet on WebStreams as the foundational primitive. Its 10 KB gzipped bundle provides a surprisingly comprehensive operator set, and the async-first design genuinely addresses real-world I/O pipeline needs.

The main areas for improvement are **dependency hygiene** (several heavy deps are underutilized or misclassified) and **safe defaults** (unbounded concurrency in parallel operations). The core architecture — TransformStream factories with a fluent chain wrapper — is sound and extensible.

**Verdict:** A solid library for its niche. The WebStreams foundation gives it natural advantages for platform interop that Observable-based alternatives lack. With dependency cleanup, the install footprint could drop from ~13.5 MB to under 2 MB with no functional loss.
